DOOGO Connect 프로젝트 성능 및 보안 리뷰 보고서
작성일: 2026-04-27 (최종 업데이트)

이 리뷰는 Next.js(App Router) 및 Supabase 기반의 아키텍처를 바탕으로 작성되었습니다.

---

### 1. 성능 (Performance) - 치명적인 순서대로 나열

1. [High] RLS 정책의 반복 계산 (InitPlan)
   - 위치: Supabase DB (admin_notices, point_purchases, profiles,   등 대다수 테이블)
   - 현상: RLS 정책에서 `auth.uid()`나 `current_setting()`을 직접 호출하여 모든 행(Row)마다 반복 계산이 발생함.
   - 영향: 데이터가 늘어날수록 조회 성능이 기하급수적으로 하락함.
   - 개선 방향: 정책 내의 함수 호출을 `(select auth.uid())`와 같이 서브쿼리로 감싸서 1회만 계산되도록 최적화 필요.

2. [High] API 페칭 워터폴 (Fetching Waterfall)
   - 위치: app/estimate/page.tsx, components/header/site-header.tsx 등
   - 현상: getSession() 호출 완료 후, 그 결과값에 의존하여 profiles, points/summary 등을 순차적으로 호출함.
   - 영향: 초기 로딩 속도가 개별 API 응답 시간의 합만큼 느려짐.
   - 개선 방향: 서버 컴포넌트(RSC)에서 데이터를 병렬(Promise.all)로 페칭하거나 클라이언트에서도 독립적인 요청은 병렬 처리해야 함.

3. [High] 클라이언트 사이드 인증 체크 및 리다이렉트
   - 위치: app/estimate/page.tsx (useEffect 내 리다이렉트 로직)
   - 현상: Middleware를 통한 서버 레벨 보호 대신, 페이지 마운트 후 클라이언트에서 세션을 확인하고 리다이렉트함.
   - 영향: 인증되지 않은 유저에게도 잠깐 동안 페이지 구조가 노출되며 불필요한 자원을 다운로드하게 됨.

4. [Medium] 거대 클라이언트 컴포넌트 (Huge Client Components)
   - 위치: app/estimate/page.tsx, components/AuthModal.tsx
   - 현상: 복잡한 비즈니스 로직과 UI가 하나의 거대 'use client' 컴포넌트에 집중됨.
   - 영향: 번들 사이즈 증가 및 리렌더링 범위 확대로 인한 인터랙션 성능 저하.

5. [Medium] 중복 보안 정책 (Multiple Permissive Policies)
   - 위치: manufacturer_container_options, manufacturer_products 등
   - 현상: 한 테이블에 동일한 권한(SELECT 등)을 허용하는 정책이 여러 개 중복 설정되어 있음.
   - 영향: DB가 권한을 확인할 때 불필요한 중복 계산을 수행하여 성능을 저하시킴.

6. [Low] 중복 인덱스 (Duplicate Index)
   - 위치: chat_rooms (chat_rooms_client_id_manufacturer_id_key, idx_chat_rooms_unique_pair)
   - 현상: 동일한 컬럼 조합에 대해 똑같은 인덱스가 2개 생성되어 있음.
   - 영향: 데이터 삽입/수정 시 DB가 인덱스를 중복 업데이트하여 쓰기 성능이 낭비됨.

---

### 2. 보안 (Security) - 위협 요소 및 수정 제안

1. [High] Service Role Client의 무분별한 사용
   - 위치: app/api/profile/sync/route.ts, app/api/referral/validate/route.ts
   - 위협: `createServiceRoleClient()`는 RLS를 완전히 우회함. 유저 권한으로 충분한 작업임에도 마스터 키를 사용하고 있어, 로직 실수 시 전체 DB 데이터가 노출/변조될 위험이 있음.
   - 제안: 반드시 필요한 경우를 제외하고는 일반 유저 세션 기반의 클라이언트를 사용하도록 변경해야 함.

2. [Low] 클라이언트 사이드 추천인 코드 유효성 검사
   - 위치: /api/referral/validate
   - 위협: 무차별 대입 공격(Brute-force)으로 유효한 파트너 코드를 수집할 수 있는 API가 공개되어 있음.
   - 제안: 해당 API에 Rate Limiting(요청 제한)을 적용하거나 인증된 유저만 호출 가능하도록 제한 권장.

### 3. 기타 권장 사항
- Supabase RLS 설정이 모든 신규 테이블에 적용되어 있는지 정기적으로 확인하십시오.
- SUPABASE_SERVICE_ROLE_KEY는 절대로 클라이언트 코드에 노출되지 않도록 서버 환경 변수로만 관리하십시오.
- `리뷰.txt`에서 해결된 항목(API 입력 데이터 검증, Middleware 도입, OTP 회원가입 로직 개선)은 본 보고서에서 삭제되었습니다.
